# Sync Memory

You are performing an end-of-session memory sync. Your job is to review the current session's
conversation and work, then persist anything valuable into the appropriate CLAUDE.md files so
future sessions start with full context.

## Step 0 — Compact auto-memory if needed

Before doing anything else, check the project's auto-memory index at
`~/.claude/projects/<project-key>/memory/MEMORY.md`.

1. Count the lines in `MEMORY.md`.
2. If the file is **150 lines or more** (approaching the 200-line truncation limit):
   - Read every memory file referenced in the index.
   - **Triage**: Score each memory on current relevance. Remove or archive memories that are:
     - About completed/resolved work (bugs fixed, features shipped, incidents closed)
     - Stale project context that no longer matches the codebase
     - Duplicated by or superseded by newer memories
     - Feedback that has been internalized into CLAUDE.md conventions
   - **Consolidate**: Merge related memories into single files where possible (e.g., three
     separate feedback memories about testing → one `feedback_testing.md`).
   - **Rewrite the index**: Update `MEMORY.md` with the compacted entries. Aim to stay under
     **120 lines** after compaction to leave headroom.
   - Delete any orphaned memory files that are no longer referenced.
3. If the file is **under 150 lines**, skip compaction and proceed.

Priority order when deciding what to keep:
1. **Feedback memories** (how the user wants to work) — almost never remove these
2. **User memories** (who the user is) — keep unless outdated
3. **Reference memories** (where to find things) — keep if the resource still exists
4. **Project memories** (ongoing work context) — most likely to go stale, prune aggressively

## Step 1 — Identify where work happened

Before reading or writing anything, determine which directories were actually worked in during
this session. Look at:

- Files that were created, edited, or read during the conversation
- Commands that were run and in which directories
- Any subdirectories of `$CWD` where meaningful work was concentrated

Build a list of **all relevant directories**. This typically includes:

- `$CWD` itself (the parent/root directory)
- Any subdirectories where files were created, modified, or where significant work took place

For example, if the user opened `~/Projects` but worked in `~/Projects/Website` and
`~/Projects/API`, all three directories may need CLAUDE.md updates — the parent for
cross-cutting context, and each subdirectory for project-specific learnings.

## Step 2 — Read all existing CLAUDE.md files

Before proposing ANY changes, read every relevant file to know what already exists:

1. **Global CLAUDE.md**: Read `~/.claude/CLAUDE.md`.
2. **CWD CLAUDE.md**: Read `$CWD/CLAUDE.md`. If it doesn't exist, note that.
3. **Subdirectory CLAUDE.md files**: For each subdirectory identified in Step 1,
   read `<subdirectory>/CLAUDE.md`. If one doesn't exist, note that you'll be creating it.

Do NOT duplicate anything that already exists in ANY of these files.

## Step 3 — Review the session

Scan the full conversation for learnings worth persisting. Look for:

- Architecture decisions or rationale
- Tool configurations, environment setup, or CLI discoveries
- Bug fixes and their root causes (the *why*, not the code)
- Workflow patterns or naming conventions established
- New dependencies, services, or integrations added
- Gotchas or non-obvious behaviors discovered
- Anything the user explicitly said to remember

Ignore ephemeral details: specific file diffs, temporary debugging steps, one-off commands
that won't recur.

## Step 4 — Route learnings to the right CLAUDE.md files

Each learning should go to the **most specific** location where it's relevant:

### Subdirectory CLAUDE.md files
Learnings that are specific to a particular subdirectory/project go into that subdirectory's
CLAUDE.md. Write directly to each subdirectory's CLAUDE.md — no approval needed. If the file
doesn't exist, create it.

### CWD (parent) CLAUDE.md
Learnings that span multiple subdirectories or apply to the parent directory as a whole go here.
Write directly — no approval needed. If the file doesn't exist, create it.

### Routing rules
- If a learning only applies to one subdirectory, put it in that subdirectory's CLAUDE.md —
  not the parent.
- If a learning applies across multiple subdirectories or to the repo/workspace as a whole,
  put it in the parent CWD CLAUDE.md.
- Never put the same learning in both a parent and subdirectory file.

### File structure
Organize under clear markdown headers. If a file already has a structure, follow it. If creating
a new one, use a structure like:

```markdown
# Project Name — CLAUDE.md

## Architecture
## Tools & Configuration
## Conventions
## Known Issues & Gotchas
## Workflows
```

Only include sections that have content. Don't create empty sections.

## Step 5 — Evaluate global-worthy items

The bar for the global `~/.claude/CLAUDE.md` is HIGH. Only propose additions that are:

- Personal workflow preferences or habits that apply across ALL projects
- Cross-project tool configurations or CLI tricks discovered
- Claude Code behaviors, commands, or settings learned
- Writing style rules or communication preferences
- Reusable patterns not tied to any single codebase
- Important personal or professional context (role changes, new clients, key decisions)

If anything qualifies:

1. **Show the user exactly what you want to add and under which section.**
2. **Wait for explicit approval before writing.**
3. Check if the global file is approaching 150 lines. If so, flag it and suggest consolidation
   before adding more.

If nothing qualifies, say so. Don't force it.

## Step 6 — Report

Give a brief summary:

- Which CLAUDE.md files were updated (and what was added to each)
- Which CLAUDE.md files were created
- What (if anything) was proposed for the global CLAUDE.md
- If nothing meaningful was learned, just say: "Nothing new to persist from this session."

## Constraints

- **Never duplicate** information already present in ANY CLAUDE.md file.
- **Never pad** files with filler content. If the session was light, say so.
- **Keep files organized** with clear markdown sections and headers.
- **Keep global CLAUDE.md lean.** Flag if approaching 150 lines.
- **Route learnings to the most specific directory possible.** Don't dump everything
  in the parent.
- **Project and subdirectory updates are direct. Global updates require approval.**
- **Always run Step 0 (auto-memory compaction check) first.** Never let MEMORY.md silently
  hit the 200-line truncation limit — compact proactively at 150 lines.
